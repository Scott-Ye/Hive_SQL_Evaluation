01 背景与目标-项目背景

课题文档给出的核心要求

  • 围绕 评测数据集搭建 与 智能响应结果模拟校验 两个核心环节展开。
  • 目标不是单次跑通，而是输出可量化、可维护、可复用的评测底座。
  • 子课题要求明确提出：要做多源自动化采集、版本演进、质量门禁与轻量解析验证链路。

课题文档参考： 课题说明原文

为什么不能用一个简单脚本解决

  • 项目要求数据集构建和评测执行分离，因此 GT、baseline、actual 不能混成一层结果。
  • 如果只有解析通过率，没有负样本和错误粒度，报告会失真。
  • 如果没有版本快照与稳定性分析，黄金集每次重跑都可能换题，无法作为长期资产复用。

→
评测策略GT / baseline / actual 分工记录

为什么“与标准答案完全一致率”不一定等于解析通过率

解析通过率只统计当前结果里有多少条被判成 pass；而“与标准答案完全一致率”统计的是当前结果是否和标准答案完全一致，既包括正样本通过，也包括负样本失败且错误粒度一致。当前这版结果中，对齐字段已经让绝大多数负样本进入严格一致统计，所以该指标提升到 94%，并明显高于解析通过率 48%。

04 难点与治理-问题与解法

难点 1：多源真实性差异

  • 问题：官方文档、benchmark、parser unit 的证据强度并不相同，不能直接混用。
  • 解决：strong / medium / weak 分层，中低可信样本进入待审池。
  • 结果：最终黄金集只保留高可信真值样本，待审池保留后续升标入口。

难点 2：版本抖动

  • 问题：同桶近似样本在不同版本间容易小范围轮换。
  • 解决：优先保留稳定核心和上一版样本，避免无谓轮换。
  • 结果：固定输入下单次运行即可收敛，版本治理重点转为“变化时如何有理由替换”。

治理结果

项目通过高可信、中等可信、低可信分层处理多源真实性差异，同时把后处理逻辑改成优先保护稳定核心和上一版样本，在仍然保留再平衡和覆盖补洞能力的前提下，避免无意义漂移。

04 难点与治理-版本治理

概念 1：初始黄金候选集

初始版本更准确地说是“初始黄金候选集”，它是从当时可获得的高可信候选池中，按结构分布、正负平衡和覆盖补洞规则选出来的第一版可用结果。

概念 2：最终黄金集

最终黄金集不是第一版结果的直接复述，而是在历史快照、稳定核心保护和必要替换规则下，逐步收敛出来的可维护版本。

概念 3：待审池

待审池不是附属产物，而是版本演进的重要输入。来源刷新、人工升标和覆盖补洞，都会优先从待审池里推动下一版变化。

来源刷新 / 升标候选池发生变化
→
读取历史快照stable_core_sql
→
优先保留稳定核心再做必要替换
→
形成下一版黄金集输出新快照
固定输入下，项目单次运行就能得到当前黄金集；当候选池发生变化时，系统会读取历史快照并优先保留稳定核心，只做有净收益的替换。 版本索引见 manifest 页面。

  • 3 条样本全部来自高可信正样本。
  • 3 条样本全部是 baseline=0、actual=0，不是本次运行新增退化。
  • 共同结论：当前 Hive Parse SDK 对部分 `WINDOW` 与 `LATERAL VIEW` 正样本仍有解析缺口。

为什么保留在黄金集里

  • 这 3 条样本都来自 Hive 官方文档，来源强、GT 强，不属于弱证据样本。
  • `WINDOW` 与 `LATERAL_VIEW` 都是当前黄金集必须保留的关键特性覆盖点。
  • 它们满足“覆盖缺口优先”，但不满足“当前解析器容易通过”。
  • 因此当前版本选择真实暴露能力缺口，而不是通过替换样本抬高通过率。

补充材料：zero_score_cases.html

样本保留判断

这 3 个 0 分样本不是版本轮换边缘造成的问题样本，而是官方文档高可信正样本中的真实解析缺口。当前项目优先保留真实来源与关键覆盖，因此没有为了提升通过率而把它们替换出去。

05 落地价值与材料-应用价值

应用价值

  • 把人工零散验 SQL 变成自动化评测流水线，提高研发、测试、回归效率。
  • 把评测集沉淀成 CSV 资产，便于协作、版本管理和下游复用。
  • GT、baseline、actual 分别记录后，评测报告更易解释，方便定位是历史对照变化还是本次解析结果变化。
  • 版本快照与待审池机制让数据集可以长期运营，而不是一次性产物。

可以继续优化的地方

  • 当前仍偏语法层评测，后续可继续补语义执行与上下文级验证。
  • 待审池升标仍要人工参与，后续可做半自动审校策略，降低人力成本。
  • 多次完整重跑会重复执行解析，可加入中间结果缓存以减少资源消耗。
  • `within_group` 等特性仍有覆盖缺口，后续应继续补强高可信真值样本。

一句话落地总结

这个项目最大的应用价值，不是单次跑出一个报告，而是沉淀了一套可以复用于后续 Parser / Agent 的评测底座：数据集可维护、评测逻辑可解释、版本治理可追踪。
